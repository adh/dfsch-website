00505 /*
00506  * Objects should be always 8-byte aligned in memory, even on 32b platforms.
00507  * When this is not true this encoding will not work. In fact, there are other
00508  * plausible representations, but introducing (backtroducing?) type pointers 
00509  * into pair is not good idea, as any overhead on pairs consumes memory, and
00510  * more importantly increases amount of cache misses.
00511  *
00512  * Object pointer tag meaning:
00513  * 000 - Normal object, first word is type
00514  * 010 - Mutable pair
00515  * 100 - Symbol
00516  * 110 - Immutable pair
00517  * 001 - Fixnum (cannot be 11, because of invalid object marker)
00518  * 101 - Small direct objects
00519  * x11 - Entry of cdr-coded list (pointer points to WORD(!), not object)
00520  *
00521  * And there are two special values of object pointer:
00522  * All zeroes - empty list
00523  * All ones - invalid object marker (e.g. end of CDR-coded list or invalid 
00524  *            weak reference, with important point is that weak-references 
00525  *            are set to this value by GC, and this is not simply changed)
00526  *
00527  * Idea is that structure of loaded code should consist of annotated pairs 
00528  * and CDR-coded sections. Immutable pairs are strictly not necessary, but 
00529  * still are good idea (for consistency and might be useful in user-code)
00530  */
